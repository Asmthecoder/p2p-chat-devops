## 🔴 CRITICAL ISSUES FIXED

### Issue #1: Messages Not Reaching Peers ❌→✅
**What was broken**: When you sent a message to a specific peer, it wasn't being delivered or stored properly.

**Why**: The `send_message()` function lost track of whether the message was broadcast or direct, and didn't properly validate which peers were actually connected.

**Fix Applied**: 
- Added proper message categorization (broadcast vs. direct)
- Improved peer connection validation  
- Better error handling for failed sends
- Messages now stored with correct context

**Impact**: Direct peer-to-peer messages now work correctly!

---

### Issue #2: Messages Not Displaying in UI ❌→✅
**What was broken**: Even when messages were stored in the database, they didn't show up in the chat window.

**Why**: 
- Message history loader showed ALL messages instead of filtering by selected peer
- Inbound message handler had inverted display logic
- History didn't reload when switching between peers
- Peer names not resolved from IDs

**Fixes Applied**:
- Selective history loading (broadcast vs. direct peer)
- Fixed message display logic
- History reloads when peer selection changes
- Peer ID → Username resolution

**Impact**: Messages now appear correctly based on which peer is selected!

---

### Issue #3: Peer Selection Broken ❌→✅
**What was broken**: Clicking on peers in the sidebar didn't work reliably.

**Why**: Used complex hidden input + button bridge pattern with closure bugs.

**Fix Applied**:
- Replaced fragile bridge with direct event handlers
- Proper lambda binding for peer IDs
- Faster peer list updates (2sec vs 5sec)

**Impact**: Clicking peers now works smoothly!

---

### Issue #4: Dialog Buttons Not Working ❌→✅
**What was broken**: "Add Peer" and "Settings" buttons crashed when clicked.

**Why**: Dialog functions were referenced before being created - classic NameError.

**Fix Applied**: 
- Moved dialog creation to beginning of build_ui()
- Proper variable scope management

**Impact**: Buttons now work without crashes!

## 📖 KEY ARCHITECTURE POINTS

**P2P Design**:
- Each peer is independent, fully autonomous
- Discovery: UDP broadcast every 30 seconds
- Communication: TCP with length-prefixed JSON protocol
- Persistence: Local SQLite database per peer
- Security: AES-256-GCM encryption on all payloads

**Fault Tolerance**:
- Heartbeat every 10 seconds
- Death detection after 3 missed beats (35 seconds total)
- Exponential backoff reconnection: 1s → 2s → 4s → ... → 30s

**Consistency**:
- Each peer maintains local message history
- No synchronization required between peers
- Eventual consistency model (perfect for P2P)
